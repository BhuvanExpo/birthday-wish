Bhuvan.live Birthday Scheduler: The Complete Architecture & Process
===================================================================

This document explains every single step of how the Bhuvan.live Birthday Scheduler functions, from the moment a user opens the webpage to the exact moment their friend receives a birthday email.

---------------------------------------------------------
PHASE 1: THE USER EXPERIENCE (FRONTEND)
---------------------------------------------------------

1. Arriving at the Website (`index.html`)
   - The user opens Bhuvan.live in their web browser.
   - The browser downloads the HTML, CSS (`style.css`), and JavaScript (`script.js`).
   - The user sees a modern, "glassmorphism" styled interface with a floating animated background.
   - They are presented with the "Schedule a Wish" form.

2. Filling out the Form
   - The user types in their Name, the Receiver's Email Address, and a custom Birthday Message.
   - They click the HTML5 date-time picker to select exactly when they want the email to be sent. The webpage's JavaScript is programmed to prevent them from picking a date in the past.

3. Submitting the Request (`script.js`)
   - The user clicks the "Schedule Email" button.
   - The JavaScript intercepts this click so the page doesn't reload.
   - The script performs "Client-Side Validation", checking that all fields are filled out correctly.
   - It takes the local date/time the user selected (e.g., exactly midnight in India) and converts it to universal UTC time so the server doesn't get confused by different timezones.
   - Finally, `script.js` packages this data into a JSON object and sends it across the internet to the backend server via a POST request to `https://birthday-wish-3zr4.onrender.com/schedule`.

4. The Feedback Prompt Loop
   - Once the server responds with "Success", the webpage shows a green checkmark notification.
   - The form resets to empty.
   - A hidden "Leave Feedback" form slides down on the page, inviting the user to tell Bhuvan what they think of the app right when they are happiest. If they submit this, it follows a similar POST request directly securely to the database.

---------------------------------------------------------
PHASE 2: THE SERVER RECEIVES THE REQUEST (BACKEND)
---------------------------------------------------------

5. The Express Server (`server.js`)
   - The server, hosted live on Render, receives the POST request.
   - First, it hits the "Rate Limiter" middleware. This checks the user's IP address. If the user has scheduled more than 100 wishes in the last hour, it blocks the request to prevent spam.
   - If allowed, it passes the data to the Validation Middleware.

6. Validation (`validationMiddleware.js`)
   - This script rigorously checks the data. It ensures the email address is formatted like a real email, that the sender name isn't a 1000-page book, and that the scheduled date is actually in the future.
   - If anything is wrong, it immediately rejects the request and sends an error back to the user's screen.
   
7. Saving to the Database (`scheduleController.js` & MongoDB)
   - If the data is perfect, the server uses a library called Mongoose to create a new `Wish` record in the MongoDB database.
   - It saves the Sender Name, Receiver Email, Message, the exact UTC date/time, and sets the initial delivery status to "pending".
   - The server sends a final "201 Created" success signal back to the user's webpage. The user's job is now done. They can close their laptop.

---------------------------------------------------------
PHASE 3: THE AUTOMATION ENGINE (CRON JOBS & NODE-CRON)
---------------------------------------------------------

8. The Relentless Loop (`server.js` Cron Job)
   - Independent of any user, the Node.js server has a "cron job" that wakes up exactly once every 60 seconds.
   - Every minute, 24 hours a day, 7 days a week, it queries the MongoDB database.
   - It asks the database: "Are there any wishes marked as 'pending' where the scheduled send time is right now (or in the past)?"
   - Most minutes, the database says "No." The cron job goes back to sleep.
   - But eventually, when the clock strikes midnight for a scheduled wish, the database replies "Yes, I have one right here."

9. Attempting Delivery
   - The server pulls the wish data from the database.
   - It reads the receiver's email and the custom message.
   - It passes this data to `emailService.js`.

---------------------------------------------------------
PHASE 4: EMAIL DELIVERY (`emailService.js`)
---------------------------------------------------------

10. Building the Email
   - The Email Service uses a library called "Nodemailer".
   - It connects via SMTP to the official email account configured for the app (e.g., Gmail or Resend).
   - It formats an email object with the subject line "Happy Birthday, [Sender's Name]!" and places the custom message inside the body.

11. Sending and Verifying
   - Nodemailer instructs the SMTP server to actually send the email over the internet to the receiver's inbox.
   - If the SMTP server confirms the email was successfully handed off to the receiver's provider (like Outlook or Gmail), it reports back "Success" to our server.
   
12. Updating the Database
   - Upon success, our server logs into MongoDB and changes the wish's status from "pending" to "sent". It will never be processed again.

13. The Fail-Safe System (Retries)
   - What if there is a network glitch and the email fails to send?
   - Instead of giving up, the server marks the wish as "failed" in the database but increments a `retryCount` by 1.
   - The next minute, when the cron job wakes up again, it specifically looks for "failed" wishes that have a `retryCount` of less than 3.
   - It will attempt to send the email again. It will try this up to 3 times to ensure temporary internet hiccups don't ruin someone's birthday surprise. If it fails 3 times, it marks it permanently failed.

---------------------------------------------------------
PHASE 5: THE ADMIN DASHBOARD (`wishes.html`)
---------------------------------------------------------

14. Viewing the Records
   - At any time, you (or a user) can navigate to the "Dashboard" page.
   - The webpage's JavaScript immediately sends a GET request to the server, asking for all scheduled wishes.
   - The server pulls the entire list from MongoDB.
   - The JavaScript receives this list and dynamically generates an HTML table row for every single wish.
   - For privacy, a function runs that visually masks the email addresses before displaying them on the screen (e.g., `friend@example.com` becomes `fri***@example.com`).
   - The table displays the sender, masked receiver, exact time it was scheduled for, and the live status ("PENDING", "SENT", or "FAILED"), providing complete transparency into the system's operation.

END OF PROCESS.
